How to configure a Samba server with SSSD in RHEL with Winbind handling AD Join

Solution Verified - Updated Friday at 8:36 PM -

Environment

  • Red Hat Enterprise Linux 7
  • Red Hat Enterprise Linux 8
  • Red Hat Enterprise Linux 9
  • SSSD
  • Samba

Issue

  • How do I configure a Samba server with SSSD in RHEL 7 or 8?

Resolution

PLEASE NOTE:

  • There are a few limitations when the idmap_sss module is used with Red Hat Enterprise Linux 7. Therefore Red Hat currently does not recommend using the idmap_sss module for Samba file server enrolled into AD domain. For further details, please see the document What is the support status for Samba file server running on IdM clients or directly enrolled AD clients where SSSD is used as the client daemon.

  • If you are configuring a RHEL 8 or above version, it is necessary to execute the command below. This command will require a reboot of the system. For more encryption types information click here.

Raw
# update-crypto-policies --set DEFAULT:AD-SUPPORT

1. Login in to server as root user.

2. Install SSSD and Samba packages.

Raw
# yum install realmd oddjob oddjob-mkhomedir sssd adcli samba samba-winbind krb5-workstation

3. Join the server using realm command.

Raw
# realm  discover testlab.redhat.com
# realm  join testlab.redhat.com -U Administrator --client-software=sssd --membership-software=samba 

In /etc/sssd/sssd.conf. In the example [domain/testlab.redhat.com] section. Add the following

ad_update_samba_machine_account_password = True

4. Start SSSD service.

Raw
# systemctl stop sssd ; rm -f /var/lib/sss/db/* ; systemctl start sssd

5. Check if AD user look and authentication is working fine.

Raw
# id <AD Username>
# kinit <AD Username>
# klist

6. Remove and Install specific packages.

Raw
# yum remove sssd-libwbclient
# yum install sssd-winbind-idmap

Note: The SSSD libwbclient implementation (i.e. the sssd-libwbclient package) has been deprecated with the release of Red Hat Enterprise Linux 8.4 and has been completely removed from Red Hat Enterprise Linux 8.5. Reference: The SSSD libwbclient implementation has been deprecated

7. Configure /etc/samba/smb.conf file as below.

Raw
[global]
    realm = TESTLAB.REDHAT.COM
    workgroup = TESTLAB
    security = ads
    kerberos method = secrets and keytab 
    template homedir = /home/%U
    idmap config * : backend = tdb
    idmap config * :  range = 10000-199999
    idmap config TESTLAB : backend = sss
    idmap config TESTLAB : range = 200000-2147483647
    load printers = no
    printing = bsd
    printcap name = /dev/null
    disable spoolss = yes
    machine password timeout = 0 

[test]
    comment = Test Share
    path = /testshare
    read only = No
    valid users = <AD Username>

8. Start the smb service.

Raw
# systemctl start smb

9. Create the test share directory.

Raw
# mkdir  /testshare
# chcon -t samba_share_t /testshare
# chmod 770 /testshare
# chown <AD Username>:<GroupName> /testshare

10. Enable and start Samba services.

Raw
# systemctl enable smb ; systemctl enable winbind ; systemctl restart smb ; systemctl restart winbind

11. Enable the Samba service in the firewall configuration.

Raw
# firewall-cmd --add-service=samba
# firewall-cmd --add-service=samba --permanent

12. Test if Samba share access is working based on AD user.

Raw
# kinit  <AD Username>
# smbclient  -L `hostname` -k
# smbclient  //`hostname`/test -k

Note: Replace all instance of <AD Username> or <GroupName> in above steps with respective user name and group.

Important

  • Using id_provider = ldap in this setup is not recommended. With id_provider = ldap it is easy to break this. Additionally id_provider = ldap does not lookup the SID of the AD objects by default which is quite important in this setup.

  • A main difference between id_provider = ldap and id_provider = ad is that the latter supports trusted domains in the local forest. A better solution here is to use id_provider = ad and the ad_enabled_domains options, see man sssd-ad for details. Finally, we strongly recommend to use id_provider = ad if you want to run a Samba file-server on the same host.

  • Since this setup is using sssd+winbind, consider also adjust the parameter ad_maximum_machine_account_password_age on /etc/sssd/sssd.conf according to system requirements, it's not a general rule, but would be good if the value does match with the same days on machine password timeout present on /etc/samba/smb.conf.

Root Cause

As of Samba 4.8, Winbind is necessary to handle the connection to the Active Directory Domain Controllers.

For additional documentation, please see the following guide:
16.1.5. Setting up Samba as a Domain Member

Joseph,

To use fully qualified names to log into the machine where SSSD is handling the authentication, you can add the following line into the [domain/...] portion of your sssd.conf:

use_fully_qualified_names = true

Afterwards, restart SSSD:

systemctl restart sssd

You should not comment out the 'account' stack where pam_sss.so sits, as this line is used to evaluate HBAC (account access) rules through SSSD.
